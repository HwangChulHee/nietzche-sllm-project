## 1. 30초 컨텍스트

**프로젝트**: 차라투스트라와의 동행 — 니체 페르소나 비주얼 노벨 (학부 캡스톤)

**무엇**: 니체 『차라투스트라는 이렇게 말했다』를 학습자가 직접 따라가는 인터랙티브 철학 소설.
- Ep 1 (8화면, 서문 1~2절 + 학습자 등장 + 동행 + 시장 도착) + Ep 2 축약 (4화면, 시장 광장 + 재회)
- 세 모드: 정적 나레이션 / 인터랙션 (차라투스트라 페르소나 sLLM) / 해설 (외부 해설자 sLLM)

**기술 스택**:
- Frontend: Next.js 16 (App Router) + TypeScript + Redux Toolkit + Tailwind
- Backend: Node + Express (`app/ml-backend/server.mjs`, .mjs ESM) + sqlite-vec + SSE 스트리밍
- LLM: Gemma 4 E2B Q4_K_M, llama.cpp 서버 (윈도우 온디바이스, 포트 8000)
- 임베딩: BGE-M3 Q4_K_M, llama.cpp 서버 (포트 8001)
- 패키징: Electron 셸 (`app/electron/`, `npm run dev`)
- 옛 FastAPI 백엔드는 `app/_archive_backend/`로 archive됨 (폐기 노선)

**현재 단계**: 옛날 상담 챗봇 (`app/`, Phase 2 완료) → 비주얼 노벨로 전환 작업 시작.

## 2. 핵심 작업 원칙

**이 4개는 절대 잊지 말 것:**

### 2.1 UI 먼저, 백엔드 나중
프론트엔드 화면 시스템부터 작동시키고, 백엔드는 Mock으로 시작. 실제 vLLM 연결은 *마지막* Phase.

### 2.2 Mock 모드 우선
모든 백엔드 엔드포인트는 처음부터 두 구현체:
- `MockLLMClient` — 미리 짠 응답 텍스트 스트리밍 (개발/시연 기본)
- `VLLMClient` — 실제 vLLM 호출 (마지막 Phase에서 swap)

환경변수 `LLM_MODE=mock` 또는 `LLM_MODE=vllm`로 토글. 기존 `app/CLAUDE.md` 패턴 그대로.

### 2.3 백엔드 인터페이스 일찍 정의
구현은 Mock이라도 *엔드포인트 스펙·요청/응답 schema*는 Phase 2에서 확정. 프론트가 그걸 호출. 나중에 Mock을 vLLM으로 swap해도 프론트 변경 X.

### 2.4 컨셉 축은 합의됨
세 모드 / 시간 흐름 / Ep 1+2 범위 / 온디바이스 비전은 *축*. 가볍게 뒤집지 말 것.
구현 디테일은 재검토 가능. UI 정책은 `VN_UI_POLICY.md`가 단일 진실 소스.

## 3. 행동 규약

### 3.1 시작 의식 (CRITICAL — 모든 Phase 시작 시)

**코드/문서 작업 전에 반드시:**

1. **현재 Phase 문서를 읽는다** (`VN_MIGRATION_PLAN.md` §해당 Phase)
2. **현재 상태를 분석하고 보고한다** — 다음 형식으로:
   ```markdown
   ## 현재 상태 분석
   
   ### 작업 영역
   - Phase: [번호 / 제목]
   - 입력 문서: [읽은 파일들]
   - 영향 범위: [건드릴 디렉토리/파일]
   
   ### 현재 코드 상태
   - 디렉토리 구조: [tree]
   - 기존 구현: [관련 파일과 기능]
   - 미구현: [없는 것들]
   
   ### 권장 작업 순서
   1. ...
   2. ...
   ```
3. 보고 출력 후 **사용자 승인 없이 바로 첫 번째 작업으로 진행**.
4. 단 §3.2의 *사용자 확인 필요 작업*에 해당하는 변경은 작업 전에 반드시 확인.

### 3.2 사용자 확인 필요 작업 (반드시 사전 확인)

다음 작업은 실행 전에 사용자에게 명시적으로 확인받을 것:

- 기존 파일 5개 이상 삭제
- 기존 디렉토리 구조 대규모 변경
- 의존성 패키지 제거 (추가는 OK)
- DB 스키마 파괴적 변경 (drop, rename)
- 환경변수 이름 변경
- README, CLAUDE.md, AGENTS.md, MIGRATION_PLAN.md 같은 문서 파일 수정
- `archived/` 디렉토리 안 파일 수정
- `ml/` 디렉토리 접근 (`.claudeignore`로 차단됨, 비주얼 노벨 작업과 무관)

### 3.3 종료 조건 명시

각 Phase 끝날 때마다:

```markdown
## Phase N 완료

### 작업한 것
- 파일1: 변경 내용
- 파일2: 변경 내용

### 검증 (어떻게 동작 확인했는가)
- [ ] 검증 항목 1
- [ ] 검증 항목 2

### `VN_PROGRESS.md`에 추가된 한 줄
- [날짜] Phase N 완료: [요약]

### 다음 Phase
- Phase N+1: [제목]
```

### 3.4 작업 스타일

- **언어**: 한국어, 짧고 핵심만, 페어 프로그래밍 톤
- **결정**: 옵션 나열 X. Claude가 합리적 판단으로 진행. 진짜 분기점만 확인 요청.
- **작업 방식**: 결과물 전에 *방향 먼저 확인* → 승인 후 진행
- **변경**: 대형 변경은 패치 단위 (수동 적용 + 커밋)
- **선호**: 비판적 피드백, 콘크리트 체크리스트/일정/결정 트리
- **스코프 관리**: *과생성 후 필터링* 철학. 리스크 높은 기능 과감히 폐기.
- **의사결정 패턴**: 구현 중 구조적 가정에 의문 제기, 더 단순한 해법으로 재방향 전환

### 3.5 절대 하지 말 것

- `ml/` 디렉토리 접근 (`.claudeignore` 차단)
- `archived/` 디렉토리 안 파일 수정 (회고용 보존)
- LangChain / LlamaIndex 도입 (직접 작성)
- 추가 LLM SDK 도입 (openai SDK만)
- 새로운 상태관리 라이브러리 (Redux Toolkit 외 추가 X)
- 인증/회원가입 시스템
- 다국어 지원
- 다크 모드 (작품 톤 깨짐)
- 모바일 반응형 (데스크톱 발표용)
- "혹시 모르니까" 만드는 기능
- 단위 테스트 작성 (시간 낭비)
- 자동 세이브
- 라우터 sLLM (학습자 명시 버튼으로 화면 전환)
- 풀이 RAG 키워드 분류기 (HyDE+점수 임계값으로 대체)

## 4. 토큰 절약 전략

각 Phase는 *별도 세션*으로 진행 권장. Phase 끝나면 `/clear` 또는 새 세션 시작.

작업 시작 시 명시적으로 파일 범위를 지정:
- *"이번엔 `services/llm_client.py`와 `core/config.py`만 작업해. 다른 파일은 읽지 마."*

작업 끝날 때 종료 조건 명확히:
- *"이 3가지가 끝나면 멈추고 보고해. 추가 작업 금지."*

긴 세션은 `/compact`로 컨텍스트 압축.
